## Architecture being specified

Marketing OS is a **hybrid deterministic/agentic orchestrator**: phase order is hardcoded in pretel-os PhaseHandlers, LLM creativity happens *inside* each handler, never in choosing the next phase. Each phase has explicit input/output JSON contracts that flow forward, plus re-trigger paths flowing backward from Phase 5.

Lifecycle (canonical artifacts in parens):
1. **Phase 0 — Research + ICP** → `product_brief_v2.json`
2. **Phase 1 — Oferta** → `offer_spec.json` + `offer_statement.md`
3. **Phase 2 — Contenido** → `content_plan.json` + `content_assets/`
4. **Phase 3 — Publicar / Distribuir** → `distribution_plan.json` + published posts
5. **Phase 4 — Medir** → `metrics_snapshot.json` (revenue attribution)
6. **Phase 5 — Ajustar / Optimizar** → `optimization_plan.json` + lessons + signal rules

Phases 0–2 are planning; 3–5 are execution + learning. Re-triggers flow backward (e.g., Phase 5 detecting avatar drift re-triggers Phase 0).

### Conceptual layers that must not be conflated

| Layer | Level | Used by |
|---|---|---|
| **ICP** | Org (B2B) / behavioral cluster (B2C) | Phase 3 targeting, lead filters |
| **Buyer Persona** | Archetypal individual within ICP | Phase 1 (Oferta) |
| **Avatar** | Contextual variant of the individual | Phase 2 (Content) |

Merging these into one entity breaks B2B and loses precision in B2C. Preserve the distinction in any spec edits.

### Runtime stack (per `specs/Overall_WF.md` §9)

- **pretel-os core**: PostgreSQL + pgvector + FastMCP server (durable state lives here)
- **Marketing OS module**: Python PhaseHandlers (SDD Planner-style) inside the pretel-os monorepo
- **LiteLLM gateway**: cascade Sonnet 4.6 → Haiku 4.5 → gpt-4o-mini
- **n8n**: external execution only (publish, scrape, webhooks); does NOT orchestrate the lifecycle
- **Storage**: `runs/{product_slug}/cycle_N_{date}/{phase}/` for run artifacts; pretel-os DB for durable state (decisions, lessons, best_practices, signal rules, lookup tables); JSON paths persisted in pretel-os via `summary_path` convention

## Hard rules from the spec (load-bearing for edits)

Explicit project commitments stated in `specs/Overall_WF.md`. Do not weaken them when editing specs:

- **Manual before automatic**: no PhaseHandler is built without the operator running that phase manually ≥3 times for real products, with each decision recorded and ≥1 best_practice distilled. (BP-001 in pretel-os, id `fea3dbd8`.)
- **Architectural honesty**: fabricated urgency/scarcity is *prohibited by the system*; risk reversal only if it would actually be honored; copy claims need evidence or are struck. These are system commitments, not operator options.
- **KPI primary = money**: soft metrics are means; Phase 4 consolidates everything to `revenue_attributed` per campaign.
- **Two distinct human decisions in Phase 0**: (1) permanent exclusions (`negative_personas`) and (2) first-cycle avatar prioritization. Neither may be algorithmic.
- **Generalization via `product_brief.json`**: never hardcode for a specific product (Declassified, etc.). New products enter through the same input contract.
- **`domain` enum constraint** (lesson `17112600`): pretel-os `best_practices.domain` only accepts `workflow|process|convention|communication`. Signal rules and lookup tables are persisted as `domain: workflow` with `SIGNAL-RULE` / `LOOKUP-TABLE` / `HOOK-TEMPLATE` prefix in title.
